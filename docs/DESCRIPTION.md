# Order Management System

## 1. Цель проекта
Собрать монолитный сервис управления заказами для интернет-магазина, который:
- принимает заказы только через API;
- обрабатывает их асинхронно в фоне;
- отправляет email-уведомления;
- предоставляет админ-панель для операционной работы команды.

Фронтенд в рамках проекта: только админ-панель.

## 2. Границы MVP
Входит в MVP:
- REST API для приема заказов.
- Хранение заказов и истории изменений статусов.
- Фоновая обработка заказов через очередь.
- Отправка email после успешной обработки.
- Админ-панель для просмотра и управления заказами.
- Ролевая модель доступа (Operator / Manager / Admin).
- Заглушка интеграции со складом.

Не входит в MVP:
- Интеграции с внешними маркетплейсами, ботами и т.п.
- Частичные отмены и частичные возвраты.
- Мультитенантность.
- Многовалютность.
- Интеграция платежей (оплата подтверждается внешней системой).
- Дополнительные каналы уведомлений (кроме email).

## 3. Основные сценарии
1. Клиентская система отправляет заказ в `POST /api/v1/orders`.
2. Система валидирует запрос и создает заказ со статусом `pending`.
3. Заказ ставится в фоновую обработку (`processing`).
4. Воркер выполняет бизнес-логику (проверки, подготовка данных для склада-заглушки).
5. При успехе заказ получает статус `completed`, отправляется email.
6. При ошибке заказ получает статус `failed` (с возможностью повторной обработки вручную).
7. Менеджер/админ может отменить заказ (`cancelled`) до завершения.

## 4. Статусы заказа
Поддерживаемые статусы:
- `pending` — заказ принят, ожидает обработки.
- `processing` — заказ в работе.
- `completed` — успешно обработан (автоматикой или ручным override).
- `failed` — завершился ошибкой.
- `cancelled` — отменен пользователем с правами.

Разрешенные переходы:
- `pending -> processing`
- `pending -> cancelled`
- `processing -> completed`
- `processing -> failed`
- `processing -> cancelled`
- `failed -> processing` (ручной ретрай)

Запрещенные переходы:
- любые переходы из `cancelled`

Примечание по `completed`:
- статус может быть установлен автоматически (воркером) или вручную (operator/manager/admin) как override;
- после установки `completed` дальнейшие переходы не допускаются.

## 5. Роли и права
Матрица доступа:

| Действие | Operator | Manager | Admin |
|---|---|---|---|
| Просмотр списка и деталей заказов | Да | Да | Да |
| Изменение статуса (`processing`, `completed`) | Да | Да | Да |
| Редактирование данных заказа (адрес, состав) | Нет | Да | Да |
| Отмена заказа | Нет | Да | Да |
| Удаление заказа (soft delete) | Нет | Нет | Да |
| Доступ к дашборду Sidekiq | Нет | Нет | Да |
| Управление пользователями и ролями | Нет | Нет | Да |

## 6. Архитектура (MVP)
Технологический подход:
- Монолит на Ruby on Rails.
- PostgreSQL как основная БД.
- Redis + Sidekiq для фоновых задач.
- Серверный HTML-интерфейс админ-панели (Rails + Tailwind CSS).

Логические блоки:
- `Order Ingestion API` — прием и первичная валидация заказов.
- `Order Domain` — модель заказа, статусы, правила переходов.
- `Background Processing` — обработка через Sidekiq/ActiveJob.
- `Notification Service` — отправка email.
- `Admin Panel` — список, фильтры, карточка, ручные действия.
- `User & Access` — Devise + Pundit, роли и политики.

## 8. API (MVP)
### Внешний API (для источника заказов)
Авторизация:
- `Authorization: Bearer <token>` для всех внешних endpoint'ов.
- токены хранятся в БД в таблице `api_tokens` только в виде хэша (не в открытом виде);
- проверка Bearer-токена выполняется на каждом запросе к внешнему API.

Идемпотентность:
- создание заказа идемпотентно по паре (`source`, `external_id`);
- при повторной отправке того же (`source`, `external_id`) система не создает дубль и возвращает уже существующий заказ.

- `POST /api/v1/orders`
  - назначение: создать новый заказ
  - ответ: `201 Created` + `order_id`, `status`
  - повторный запрос с той же парой (`source`, `external_id`): `200 OK` + существующий `order_id`, `status`
- `GET /api/v1/orders/:id`
  - назначение: получить текущий статус заказа
  - ответ: `200 OK` + данные заказа

Примечание: отдельный публичный API для широкого внешнего чтения заказов в MVP не нужен. Текущий `GET /orders/:id` ограничивается задачей проверки статуса по конкретному заказу.

### Админ API/контроллеры (внутренние)
- список заказов (фильтры по статусу, дате)
- просмотр деталей заказа
- изменение статуса (с проверкой допустимых переходов)
- редактирование данных заказа (Manager/Admin)
- отмена заказа (Manager/Admin)
- удаление заказа (soft delete, Admin)

## 9. Очереди и фоновые задачи
Основные джобы:
- `ProcessOrderJob` — обработка заказа.
- `SendOrderCompletedEmailJob` — отправка email после `completed`.

Поведение обработки:
- задача обработки запускается после создания заказа;
- при ошибке обработки заказ переводится в `failed`;
- для `failed` доступен ручной повторный запуск из админ-панели;
- отправка email выполняется отдельной задачей после успешного завершения;
- при ручном override в `completed` email также отправляется.

## 10. Интерфейс админ-панели
Разделы MVP:
- Страница входа.
- Лента заказов (таблица, фильтры, быстрые действия).
- Карточка заказа (состав, клиент, адрес, история статусов).
- Экран управления пользователями (только Admin).
- Переход в Sidekiq dashboard (только Admin).

Поведение soft delete:
- soft deleted заказы скрыты из списка по умолчанию;
- для Admin доступен фильтр `show deleted` для просмотра удаленных заказов.

## 12. План реализации
### Этап 1. Базовый каркас
- Инициализация Rails-проекта и инфраструктуры Docker Compose.
- Подключение PostgreSQL/Redis/Sidekiq.
- Настройка RSpec, RuboCop, базового CI.

### Этап 2. Домен заказов
- Миграции `orders`, `order_items`, `order_status_events`.
- Модель статусов и валидация переходов.
- Сервис создания заказа из API-запроса.

### Этап 3. API заказов
- `POST /api/v1/orders`.
- `GET /api/v1/orders/:id`.
- Тесты API-контрактов и валидаций.

### Этап 4. Фоновая обработка
- `ProcessOrderJob` и заглушка складской интеграции.
- Логика переходов `pending -> processing -> completed/failed`.
- Механизм ручного ретрая для `failed`.

### Этап 5. Email-уведомления
- Почтовый шаблон и `SendOrderCompletedEmailJob`.
- Запуск только после `completed`.
- Тесты отправки и защиты от повторных отправок.

### Этап 6. Админ-панель и роли
- Аутентификация Devise.
- Авторизация Pundit по матрице ролей.
- Страницы: список, детали, редактирование, отмена, удаление.
- Экран пользователей и назначение ролей (Admin only).

### Этап 7. Финализация MVP
- Подключение Sidekiq dashboard с ограничением по роли.
- Сквозные сценарии (создание -> обработка -> email -> действия менеджера).
- Подготовка deploy-инструкции на VPS.

## 13. Критерии готовности MVP
MVP считается готовым, когда:
- API принимает и сохраняет корректные заказы.
- Внешний API защищен Bearer-токеном.
- Повторный `POST` с тем же (`source`, `external_id`) не создает дубль (идемпотентность).
- Заказы проходят фоновую обработку и получают финальные статусы.
- Email отправляется при успешном завершении заказа.
- В админ-панели доступны операции в соответствии с ролями.
- Admin может управлять пользователями и ролями.
- Система развертывается на VPS по инструкции проекта.
